# Versioning Policy

Status: Draft  
Applies to: DASObjectStore source, CLI, service contracts, and persistent
metadata formats

## Semantic Versioning

DASObjectStore SHALL use semantic versioning once releases begin.

Version numbers follow:

```text
MAJOR.MINOR.PATCH
```

- `PATCH` releases contain compatible fixes only.
- `MINOR` releases add backward-compatible behavior.
- `MAJOR` releases may introduce breaking changes.

## Pre-1.0 Compatibility

Before `1.0.0`, DASObjectStore may change internal implementation details more
freely, but it should still treat the following as compatibility-sensitive:

- persistent metadata schemas;
- pool manifests;
- disk manifests;
- append-only placement logs;
- CLI command names and machine-readable output;
- store policy keys and semantics;
- object lifecycle state names;
- service provider configuration generated for users;
- Mnemosyne/Mneion export formats.

Breaking changes to those surfaces must be documented in the commit, release
notes, or an ADR before implementation lands.

## Public Compatibility Surfaces

The following surfaces are public once documented or shipped in a release:

- `dasobjectstore` CLI commands;
- `--json` output schemas;
- configuration file formats;
- metadata manifest formats;
- S3/object-service orchestration layout generated by DASObjectStore;
- read-only export behavior;
- Mnemosyne adapter exports.

Rust crate APIs are not considered stable until explicitly documented as public
library APIs.

## Persistent Metadata Format Versions

Persistent metadata must carry explicit format versions.

At minimum, these formats require independent versions:

- live SQLite metadata schema;
- canonical pool manifest;
- canonical disk manifest;
- append-only placement log;
- store policy document;
- Mnemosyne export document.

Metadata readers must reject unknown future major versions rather than guessing.

Metadata readers may accept older compatible versions through explicit
migrations.

## Migration Rules

Metadata migrations must be:

- explicit;
- tested;
- recoverable from backup or snapshot when practical;
- documented when they affect user recovery behavior.

Destructive or lossy migrations must require explicit user confirmation.

Automatic migrations must not run during read-only attach or read-only dirty
import.

## CLI Compatibility

Human-readable CLI output may evolve between minor releases.

Machine-readable CLI output, especially `--json`, is compatibility-sensitive and
must include enough versioning or command context for downstream tools to detect
schema changes.

Risky command semantics, such as force import, force retire, repair, drain, and
direct-to-HDD import, must not change silently.

## Release Readiness

A release that changes compatibility-sensitive behavior should update at least
one of:

- this document;
- `ROADMAP.md`;
- `TODO.md`;
- an ADR or design note;
- release notes.
